I’m looking at this as a data/modeling problem rather than a charting one: the key bit is deciding what volatility definition you want, then storing or calculating it consistently so your existing charts can plot it.

Yes — this is very doable.

For a price series, “volatility” usually means how much returns move around, not just how much price changes. So the normal approach is:

  1. calculate daily returns from price_histories.price

  2. take a rolling standard deviation of those returns over a window like 21 days, 30 days, 90 days

  3. optionally annualise it

The most common formula is:

  • daily return:
    r_t = LN(price_t / price_t-1)
    or simple return:
    (price_t / price_t-1) - 1

  • rolling volatility over N days:
    STDDEV(r_t over last N days)

  • annualised volatility:
    rolling_stddev * SQRT(252)

252 is used because there are about 252 trading days in a year.

Because you already have price_histories with columns like MA_5d, MA_40d, VOL_90d, the cleanest setup is:

  • plot them as a separate line chart

How to calculate it in SQL

If you are using MySQL 8+, you can do this with window functions.

Step 1: calculate log returns

SQL
WITH returns AS (
  SELECT
    security_id,
    price_date,
    price,
    LN(price / LAG(price) OVER (
      PARTITION BY security_id
      ORDER BY price_date
    )) AS log_return
  FROM price_histories
)
SELECT *
FROM returns;

Step 2: calculate rolling volatility

For 21-day vol:

SQL
WITH returns AS (
  SELECT
    security_id,
    price_date,
    LN(price / LAG(price) OVER (
      PARTITION BY security_id
      ORDER BY price_date
    )) AS log_return
  FROM price_histories
)
SELECT
  security_id,
  price_date,
  STDDEV_SAMP(log_return) OVER (
    PARTITION BY security_id
    ORDER BY price_date
    ROWS BETWEEN 20 PRECEDING AND CURRENT ROW
  ) * SQRT(252) AS vol_21d
FROM returns;

For 90-day vol:

SQL
WITH returns AS (
  SELECT
    security_id,
    price_date,
    LN(price / LAG(price) OVER (
      PARTITION BY security_id
      ORDER BY price_date
    )) AS log_return
  FROM price_histories
)
SELECT
  security_id,
  price_date,
  STDDEV_SAMP(log_return) OVER (
    PARTITION BY security_id
    ORDER BY price_date
    ROWS BETWEEN 89 PRECEDING AND CURRENT ROW
  ) * SQRT(252) AS vol_90d
FROM returns;

Updating your table

You can use an update join like this:

SQL
WITH returns AS (
  SELECT
    security_id,
    price_date,
    LN(price / LAG(price) OVER (
      PARTITION BY security_id
      ORDER BY price_date
    )) AS log_return
  FROM price_histories
),
vols AS (
  SELECT
    security_id,
    price_date,
    STDDEV_SAMP(log_return) OVER (
      PARTITION BY security_id
      ORDER BY price_date
      ROWS BETWEEN 89 PRECEDING AND CURRENT ROW
    ) * SQRT(252) AS vol_90d
  FROM returns
)
UPDATE price_histories ph
JOIN vols v
  ON ph.security_id = v.security_id
 AND ph.price_date = v.price_date
SET ph.VOL_90d = v.vol_90d;

If your MySQL setup complains about WITH ... UPDATE, do it as a temp table first.

Better option: calculate in Python backend

Honestly, for your app this is often easier and cleaner in Python/pandas than SQL.

Example:

Python
import pandas as pd
import numpy as np

df = df.sort_values(["security_id", "price_date"])

df["log_return"] = df.groupby("security_id")["price"].transform(
    lambda s: np.log(s / s.shift(1))
)

df["VOL_21d"] = df.groupby("security_id")["log_return"].transform(
    lambda s: s.rolling(21).std() * np.sqrt(252)
)

df["VOL_90d"] = df.groupby("security_id")["log_return"].transform(
    lambda s: s.rolling(90).std() * np.sqrt(252)
)

Then write those values back into price_histories.

That is usually the easiest route if you already have a data pipeline updating MAs.

How to display it on a chart

Two common choices:

Option 1: separate volatility chart

Best option.

  • top chart = price

  • bottom chart = volatility

Why this is better:

  • price and vol are on very different scales

  • overlaying them usually looks messy

Option 2: overlay on secondary y-axis

Possible, but less clean.

# db_utils.py

from datetime import date

def get_seasonality_chart_by_security_id(sec_id, start_date, connection):
    """
    Calls stored procedure:
    get_seasonality_chart_by_security_id(sec_id INT, start_date DATE)
    """
    cursor = connection.cursor(dictionary=True)

    try:
        cursor.callproc("get_seasonality_chart_by_security_id", [sec_id, start_date])

        results = []
        for result in cursor.stored_results():
            results = result.fetchall()

        return results

    finally:
        cursor.close()

@routes.route("/seasonality/<int:security_id>", methods=["GET"])
def seasonality_chart(security_id):
    if security_id not in ALLOWED_SEASONALITY_SECURITIES:
        return jsonify({"error": "Security not allowed for seasonality view"}), 400

    start_date = request.args.get("start_date", "2010-01-01")

    try:
        datetime.strptime(start_date, "%Y-%m-%d")
    except ValueError:
        return jsonify({"error": "Invalid start_date format. Use YYYY-MM-DD"}), 400

    connection = get_db_connection()

    try:
        data = get_seasonality_chart_by_security_id(security_id, start_date, connection)
        return jsonify(data), 200
    except Exception as e:
        return jsonify({"error": str(e)}), 500
    finally:
        connection.close()

    ]

    connection = get_db_connection()
    cursor = connection.cursor(dictionary=True)

    try:
        placeholders = ",".join(["%s"] * len(allowed_ids))
        query = f"""
            SELECT
                s.security_id,
                s.security_long_name,
                s.security_short_name,
                ac.asset_class_name
            FROM securities s
            LEFT JOIN asset_classes ac
                ON s.asset_class_id = ac.asset_class_id
            WHERE s.security_id IN ({placeholders})
            ORDER BY ac.asset_class_name, s.security_long_name

        cursor.execute(query, allowed_ids)
        return cursor.fetchall()

    except Exception as e:
        logging.error(f"Error fetching seasonality securities: {e}")
        raise

    finally:
        cursor.close()
        connection.close()

          <div className="seasonality-chart-wrapper">
            <Line data={lineData} options={options} />
          </div>
        ) : (
          <div className="seasonality-message">
            Select an asset class and security to load seasonality data.
          </div>
        )}
      </div>
    </div>
  );
};

7) one likely issue in your stored procedure

Your stored procedure currently has:

SQL
MAKEDATE(2023, doy) AS calendar_date

That is fine for plotting a Jan-to-Dec seasonal curve, but be aware:

  • 2023 is arbitrary

  • for leap-year edge cases, you are already excluding doy > 365, so that part is okay

That means the frontend is not plotting actual historical years, just the synthetic seasonal year. Which is exactly what you want for seasonality.

8) one thing to check in your schema

This query assumes you have an asset_classes table with:

  • asset_class_id

  • asset_class_name

because your securities table only has asset_class_id.

If your actual asset class table/column names are different, change this:

SQL
LEFT JOIN asset_classes ac
    ON s.asset_class_id = ac.asset_class_id

and:

SQL
ac.asset_class_name

to your real names.
